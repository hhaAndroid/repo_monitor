## 1. 项目总体需求

给定某个 GitHub repo 地址，定时监控 main 分支合入 PR，将符合用户需求的 PR 记录到特定的文档中，方便跟踪 repo 进展。

---

## 2. 核心目标

- ⏰ **定时监控**: 每天 8:30 自动监控 GitHub repo，监控 main 分支合入 PR
- 🎯 **精准筛选**: 基于预定义需求精确筛选，只关注重要特性
- 📝 **格式清晰**: 新建的 PR 追踪文档要格式排版好看清晰
- 📂 **结构清晰**: 每个仓库独立文档，方便查看和管理
- 🔄 **支持恢复**: 记录文档中要记录当前监控到的最新 PR 是哪一条，下一次查看时候不要重复监控
- 📊 **首次监控**: 如果某个 repo 是第一次监控，则第一次仅仅查看最新的 30 个合入 PR 即可
- 🔗 **跳转链接**: PR 追踪文档里面记录的 PR 信息要包括跳转链接，否则不好直接找到对应 PR
- 📱 **消息通知**: 每天定时追踪时候，如果有新的更新内容就发消息通知我，如果没有任何更新则直接发消息告诉我：今天 xxx 库没有更新

## 4. PR 筛选规则

### 4.1 必须关注（高优先级）

| 类型 | 关键词 | 优先级 | 说明 |
|------|--------|--------|------|
| 🚨 BREAKING | `BREAKING` | P0 | 破坏性变更 |
| ✨ 新功能 | `feat:` | P0 | 新增算法、功能 |
| 🔧 重构 | `refactor:` | P1 | 代码重构 |
| ⚡ 性能 | `perf:` | P1 | 性能优化 |

### 4.2 排除（不记录）

| 类型 | 关键词 | 说明 |
|------|--------|------|
| 🐛 Bug 修复 | `fix:` | 一般 bugfix |
| 📝 文档 | `doc:` | 文档更新 |
| 🔧 杂项 | `chore:`, `ci:` | 构建/配置更新 |
| 🧪 测试 | `test:` | 测试相关 |

## 6. 首次监控流程

1. 查询仓库最新 30 个已合并 PR
2. 按筛选规则过滤（feat/refactor/BREAKING）
3. 创建仓库对应的 Markdown 追踪文档
4. 记录最新 PR 编号
5. 生成统计汇总

---

## 7. 定时监控流程（每天 8:30）

1. 读取上次记录的最新 PR 编号
2. 查询该编号之后的新 PR
3. 筛选符合要求的 PR
4. **如果有新 PR**:
   - 追加到追踪文档
   - 更新最新 PR 编号
   - 发送消息通知："xxx 库有 X 个新 PR"
5. **如果没有新 PR**:
   - 发送消息："今天 xxx 库没有更新"

## 8. PR 追踪文档格式

每个仓库的追踪文档使用以下格式：

```markdown

> 最后更新: YYYY-MM-DD  
> 最新 PR: [#XXXX](链接)

## 📌 项目信息

- **仓库地址**: https://github.com/xxx/xxx
- **监控开始**: YYYY-MM-DD

## 🔥 重要 PR 记录

### YYYY-MM-DD 更新

#### 🔴 BREAKING / 🟡 新功能 / 🔧 重构

**[#XXXX](链接)** `类型: 标题`
- **作者**: xxx
- **说明**: xxx
- **价值**: xxx

## 📊 统计

| 类型 | 数量 |
|------|------|
| BREAKING | X |
| 新功能 | X |
| 重构 | X |

## 📝 监控记录

- **首次监控**: YYYY-MM-DD
- **最新 PR**: #XXXX
- **下次监控**: YYYY-MM-DD 08:30
```
